## 1. Context and motivation

The plugin's compound-learning loop is `reflections/active/ → REFLECTION_LOG.md → human curates
→ AGENTS.md / HARNESS.md`. Today, a human reading the aggregate must do *all* the cognitive
work of clustering recurring signals, deciding which are worth a rule, and judging whether a
proposed rule would actually have helped. As the corpus grows, that intake is the bottleneck:
the HARNESS "Reflection-driven regression detection" GC rule already exists precisely because
recurring failure patterns hide in the log, unpromoted.

Umbrella **D6** raises the *proposal* quality of this loop **without touching the human-curates
principle**. A dynamic workflow (`reflection-mining.workflow.js`, shipped in S2):

1. **Clusters** the reflection entries with parallel agents — grouping by the underlying
   recurring theme, one candidate rule per cluster (generate).
2. **Adversarially pre-filters** each candidate: a skeptic agent, judging each candidate in its
   own context, asks *"would this proposed rule actually have prevented a real past mistake
   recorded in the corpus?"* and refutes it if not (filter).
3. Emits a **vetted shortlist** of promotion candidates — each with its proposed rule, the
   source fragment(s) it clusters, and the adversarial-filter verdict + evidence — to a
   **staging artefact, `REFLECTION_STAGING.md`** (shortlist).

The pattern is **generate-and-filter + adversarial verification** — the *reverse* of the
rule-adherence pattern: over-generate candidate rules, then prune hard.

**The load-bearing boundary (INV-1).** Mining is **propose-only**. It writes **only** to
`REFLECTION_STAGING.md` — a non-durable working artefact — and **never** to `AGENTS.md`,
`HARNESS.md`, `CLAUDE.md`, or `MODEL_ROUTING.md`. `AGENTS.md` is **byte-for-byte unchanged** by
any `--mine` run. A human reads the staging shortlist and promotes what they choose through the
**existing `Promoted:` flow** (a `Promoted:` line on the source fragment, in the same commit as
the `AGENTS.md`/`HARNESS.md` edit). This is "agents propose; humans curate" applied to the
learning loop itself: mining produces a *better-vetted proposal*; the human-curation gate is
**unchanged**.

The slicing record records the `decision-boundary` lens: the staging-location choice (a new
`REFLECTION_STAGING.md`, resolved at open-question 3) reshapes the GC/snapshot/ergonomics
surface downstream without being a mere implementation detail. §6 sequences S6 last among
behavioural changes — it augments the learning loop once the workflow-mode patterns are mature.

## 3. User story

> **As a** curator maintaining the project's durable theory of the system (AGENTS.md /
> HARNESS.md), **I want** to run `/reflect --mine` to have a dynamic workflow cluster the
> reflection corpus and adversarially pre-filter candidate rules — keeping only those that
> *would have prevented a real past mistake* — into a vetted shortlist in `REFLECTION_STAGING.md`,
> **so that** I can promote high-value rules with the clustering-and-vetting work already done,
> **while** the human-curation gate stays the only path into AGENTS.md: mining proposes, I
> still curate, and AGENTS.md is byte-for-byte unchanged until I promote.

This is the umbrella's **INV-1** held load-bearing in the learning loop: mining raises proposal
quality without ever writing a durable artefact; the human `Promoted:` flow is unchanged.

## 5. The `REFLECTION_STAGING.md` artefact contract

Resolved (open-question 3): the staging artefact is a **new file, `REFLECTION_STAGING.md`** —
clean separation from the append-only fragment store and its generated aggregate.

### 5.1 Shape

A staging shortlist of promotion candidates. Each candidate carries, at minimum:

- **Proposed rule / insight** — the candidate rule the cluster yields (one per cluster).
- **Source reflection fragment(s)** — the `reflections/active/<date>-<slug>.md` fragment(s) the
  candidate clusters (so the human can trace it back to the evidence).
- **Adversarial-filter verdict** — the answer to *"would this rule have prevented a real past
  mistake recorded in the corpus?"* plus the supporting **evidence** (which fragment/mistake it
  would have caught). Only candidates the skeptic affirms reach the shortlist.

(The exact field layout is an implementation choice for the tdd/impl phase; this contract fixes
*what each candidate must carry*, not a frozen markdown schema. The S2 template's
`CANDIDATE_SCHEMA` — `rule`, `wouldHavePrevented`, `evidence` — is the authoritative field set
the mode adapts.)

### 5.2 Status and lifecycle (resolved — §6 decision 1)

- `REFLECTION_STAGING.md` is a **staging / working** area for human review — **NOT** the
  permanent record and **NOT** a durable curated artefact in the INV-1 sense. It is the
  mining-loop analogue of the `reflections/active/` working set (transient) versus
  `REFLECTION_LOG.md` (the committed aggregate) — except staging is a *proposal scratchpad*, not
  a record at all.
- A human reads it, promotes what they choose into `AGENTS.md`/`HARNESS.md` via the existing
  `Promoted:` flow (a `Promoted:` line on the **source fragment**, in the same commit as the
  durable edit), then the staging entry can be **cleared**.
- **`AGENTS.md` is byte-for-byte unchanged by mining** (umbrella D6 deterministic acceptance) —
  mining writes only the staging file.

### 5.3 The INV-1 firewall position

`REFLECTION_STAGING.md` is **not** one of the four durable curated artefacts the INV-1 firewall
guards (`HARNESS.md`, `AGENTS.md`, `CLAUDE.md`, `MODEL_ROUTING.md`). Writing the staging file is
therefore *permitted* — it is the propose half of "agents propose; humans curate". The S2
`reflection-mining.workflow.js` already routes its shortlist to a **non-durable staging sink**
(`stagingSink`, defaulting to `REFLECTION_STAGING.md`) and names the durable artefacts in its
preamble only to document the boundary, never reaching them in code; the `inv-firewall.sh` grep
over `*.workflow.js` already passes for it. S6 adds no new firewall surface — it confirms the
staging file is the *only* write `--mine` makes.

## 6. Decisions surfaced at the GATE

Per the AGENTS.md STYLE note, a recommendation is a **hypothesis for the spec-mode diaboli to
stress**, not a default to confirm — most acutely where the slice touches a **closed contract**
(here: the INV-1 four-artefact denylist; the human-curates `Promoted:` flow; the reflection
storage model and its `regenerate-reflection-log.sh` generator). The three decisions below are
surfaced for the human; the spec-writer recommends but does not silently choose.

**Pre-listed contract surfaces a recommendation could break** (so the diaboli's scrutiny is
scoped): (1) the INV-1 four-artefact denylist and firewall — mining must never write a durable
artefact; (2) the human `Promoted:`-line curation flow — mining must not bypass or auto-promote;
(3) the reflection storage model (`reflections/active/` fragments → `REFLECTION_LOG.md`
aggregate via `regenerate-reflection-log.sh`) — the staging file must not collide with the
fragment store, the aggregate, or the regenerator; (4) the default `/reflect` capture path — it
must stay byte-identical in behaviour.

### Decision 1 — `REFLECTION_STAGING.md` lifecycle (committed vs gitignored; regenerated vs accumulate; where it lives)

Open-question 3 is **resolved to a new file**; what remains is **how it lives**. Mirroring the
reflections working-set/aggregate distinction:

- **Option L1 — gitignored, regenerated-each-run working file at the repo root (RECOMMENDED).**
  `REFLECTION_STAGING.md` lives at the **repo root** (alongside `REFLECTION_LOG.md`, so the
  curator finds it where they already look), is **gitignored**, and each `--mine` run
  **regenerates** it (overwrite, not accumulate). *Strength:* it is a **proposal scratchpad**,
  not a record — committing it would (a) put ephemeral, churny mining output into the durable
  history the §7 governance-erosion risk warns against, and (b) blur the staging/permanent line
  the contract draws. The plugin already gitignores exactly this class of artefact — per-session
  drafts reviewed and copied into a durable file by hand (`.claude/affordance-discovery-*.md`,
  `cost-estimates/`, `diagnostic-legibility/output/`). A staging shortlist is the same kind of
  thing: **derived, regenerable, human-reviewed, then cleared.** Regenerate-each-run keeps it
  honest — it always reflects the *current* corpus, never a stale accumulation. *Diaboli should
  stress:* does gitignoring lose an audit trail the team needs (no — the **promotion** is the
  durable record, captured by the `Promoted:` line on the source fragment; the *proposal* need
  not be archived)? Does the repo-root location collide with anything (no — `REFLECTION_STAGING.md`
  is a new name; checked, no existing reference)?
- **Option L2 — committed, regenerated-each-run, repo root.** *Weakness:* commits ephemeral
  proposal churn to durable history and would invite a "Reflections via PR workflow"-style
  constraint and a GC/staleness rule the v1 does not need (cf. AGENTS.md "no threshold before
  data"). The §7 governance-erosion risk treats ephemeral churn entering the durable surface as
  the thing INV-1 exists to prevent. *Tenable only* if the team wants the proposal history
  itself preserved — but that duplicates what the `Promoted:` line already records durably.
- **Option L3 — committed, append/accumulate.** *Rejected:* an accumulating committed file grows
  unboundedly with superseded proposals, needs its own archival GC, and diverges hardest from
  the "staging scratchpad, then cleared" contract. It re-creates the very intake-bloat problem
  mining exists to reduce.

**Recommendation: L1** — `REFLECTION_STAGING.md` at the **repo root**, **gitignored**,
**regenerated (overwritten) each `--mine` run**. It mirrors the plugin's established
gitignored-working-draft pattern (affordance-discovery, cost-estimates, diagnose output),
keeps ephemeral proposal churn out of the durable history INV-1 protects, and keeps the staging
file honest to the current corpus. **Add a `.gitignore` entry** for `REFLECTION_STAGING.md`
(grouped with the other derived/regenerable working artefacts) — this is the one .gitignore
touch S6 makes. No GC rule, no constraint, no archival surface in v1 (flagged in §9).

### Decision 2 — `--mine` fallback without the Claude Code runtime

This is the per-slice instance of the umbrella runtime-scope fallback (§5.5). The broader
Copilot degradation *contract* (guidance-only vs omit the skill) is open-question 4, dispositioned
in S7; S6 must still state its **own local behaviour**.

- **Option F1 — guidance-only degradation (RECOMMENDED).** On a tree without the workflow
  runtime, `--mine` does **not** spawn a workflow and does **not** error. Instead it prints a
  short guidance message: that mining requires the Claude Code dynamic-workflows runtime, that
  the knowledge (cluster → adversarially filter → shortlist) is still readable in the
  `dynamic-workflows` skill, and that the curator can do the clustering/vetting by hand and
  promote through the existing `Promoted:` flow. *Strength:* matches the umbrella's settled
  "degrade to guidance only — knowledge stays readable, no workflow spawned, never error"
  direction (§5.5) and the S3/S4/S5 fallback shape; the curator is left able to act, just
  manually. *Diaboli should stress:* is "guidance" overclaiming relative to S7's still-open
  contract (no — S6 states only its *local* non-erroring guidance-only behaviour; S7 fixes the
  cross-CLI documentation/omit contract)?
- **Option F2 — a no-op that explains.** `--mine` prints "mining needs the Claude Code workflow
  runtime; not available here" and exits without further guidance. *Weakness:* thinner than F1 —
  it explains *why* nothing happened but does not point the curator at the readable knowledge or
  the manual path, leaving the curator stranded. F2 is essentially F1 minus the constructive
  pointer.

**Recommendation: F1** — guidance-only, non-erroring: explain that `--mine` needs the Claude
Code runtime, point at the readable `dynamic-workflows` skill knowledge and the manual
cluster/vet/promote path, never error. The default `/reflect` capture path is unaffected on
every tree. This is the umbrella §5.5 fallback applied locally; S7 still owns the precise
cross-CLI contract.

### Decision 3 — The verification split (deterministic structural vs agent-backed/observational)

Mirroring S3 decision 2 / S4 decision 2 / S5 decision 2: this repo's deterministic layer
**reads files and matches structure**; it does **not** dispatch `/reflect --mine`, run a
workflow, or diff `AGENTS.md`. So:

- **Deterministically assertable (structural, Layer-0/1) — the declarations.** That `reflect.md`
  **declares**: the `--mine` mode exists and is optional, default capture path unchanged (AC-3);
  the cluster → filter → shortlist shape + adapt-by-relative-path (AC-4); the INV-1 staging-only
  / never-AGENTS.md / byte-for-byte-unchanged / human-promotes boundary (AC-5); the staging
  candidate shape + staging-not-permanent status (AC-6); the Claude-Code-only scope +
  non-erroring guidance-only fallback (AC-7). That `integration-agent.agent.md` **declares**
  mining augments-never-replaces human curation (AC-8). And the **AGENTS.md-immutability triad**
  (AC-2): the two doc declarations above plus the already-passing `inv-firewall.sh` over the S2
  template. These are file-read assertions a structural test checks mechanically — the **honest
  deterministic shadow** of D6.
- **Agent-backed / behavioural only (not deterministic) — the live mining run.** That a real
  `--mine` run **clusters the corpus, adversarially filters, writes the shortlist to
  `REFLECTION_STAGING.md`, and leaves `AGENTS.md`'s bytes identical** (AC-1) is a runtime
  property of an actual workflow dispatch. It is **agent-backed**, observed at runtime, and must
  **not** be promised as deterministic.
- **Observational / unverified (AC-9)** — the **GC-pressure reduction** is an observation to
  record in a snapshot / first-run note, not a file-read or counter. Must **not** be
  over-promised as deterministic.

**Recommendation:** the tdad-scenario set is **predominantly structural** (AC-2 through AC-8 as
Layer-0/1 file-read assertions), with **AC-1** standing as a **declared agent-backed**
behavioural scenario and **AC-9** as **observational/unverified**. This is the same realistic
split S3/S4/S5 landed. The tdd-agent will create
`tdad_tests/scenarios/commands/reflect/` (and `tdad_tests/scenarios/agents/integration-agent/`
for AC-8), author the structural scenarios (AC-2 through AC-8), write a new deterministic content
test `tdad_tests/tests/test_s6_reflection_mining_structural.py` (mirroring
`test_s5_orchestrator_routing_structural.py` — a section-slicing helper isolating the `--mine`
section in `reflect.md` and the augments-not-replaces note in `integration-agent.agent.md`,
phrase assertions per AC), and **wire it into the existing `.github/workflows/tdad-tests-fast.yml`**
as a new step (a sibling of the existing `Run Layer 1 (dynamic-workflows S5 orchestrator
routing)` step at lines 72–74). Spec-writer **names** these homes; spec-writer does **not**
create the directories or any test file (spec-first discipline — no test files in this commit).

The diaboli should stress whether any property tagged structural actually requires a live
dispatch (it should not — each is a file read), and whether AC-2's "byte-for-byte unchanged"
risks being over-promised as deterministic (it must not — the *declarations + firewall* are
structural; the *live byte-identity* is the agent-backed half of AC-1).

## 9. Risks and open questions

**Risks.**

- *INV-1 erosion (the critical risk for this slice).* The temptation is to let mining write a
  rule straight into `AGENTS.md`. Mitigation: `--mine` writes **only** `REFLECTION_STAGING.md`
  (a non-durable staging file); AC-5/FR-3 declare the staging-only / never-AGENTS.md boundary;
  the S2 template's `inv-firewall.sh` already forbids durable writes in the workflow code; AC-2's
  triad witnesses the guarantee deterministically. The diaboli should reject any wording that
  lets mining promote, auto-curate, or write a durable artefact.
- *Eroding the human-curates principle.* Mining is a *better proposal*, not a curator. Mitigation:
  AC-8/FR-6 make the integration-agent state mining augments-never-replaces curation; the human
  `Promoted:` flow is the only path into `AGENTS.md`/`HARNESS.md` and is unchanged.
- *Ephemeral churn entering durable history.* A committed/accumulating staging file would push
  ephemeral proposal churn into the durable surface INV-1 protects. Mitigation: §6 decision 1's
  L1 — gitignored, regenerated-each-run — keeps the proposal a scratchpad; the **promotion** is
  the durable record (the `Promoted:` line on the source fragment).
- *Over-promising determinism.* The live mining run (AC-1) and the GC-pressure reduction (AC-9)
  are agent-backed/observational; tagging them deterministic would overclaim. Mitigation: §6
  decision 3 fixes the honest split, as in S3/S4/S5; AC-2's "byte-for-byte unchanged" is asserted
  as a structural *triad*, not a live diff.
- *Default-path regression.* The `--mine` addition must not change bare `/reflect` capture.
  Mitigation: AC-3/FR-1 declare `--mine` additive and the capture path unchanged; the diaboli
  should reject any edit that touches the default `## Process` behaviour.
- *Staging-file collision with the reflection store.* `REFLECTION_STAGING.md` must not collide
  with `reflections/active/`, `REFLECTION_LOG.md`, or `regenerate-reflection-log.sh`. Mitigation:
  it is a **new, distinct** repo-root file (checked — no existing reference) that the regenerator
  neither reads nor writes.

**Open questions.** None block S6. The umbrella open question Q4 (Copilot degradation contract)
rides **S7** and is out of scope (§2.2) — S6 states only its own local guidance-only fallback
(§6 decision 2). **Open-question 3 (staging artefact) is resolved: a new `REFLECTION_STAGING.md`**.
The only S6-internal decisions are the staging-file **lifecycle** (§6 decision 1), the `--mine`
**fallback** (§6 decision 2), and the **verification split** (§6 decision 3) — all **surfaced for
the GATE, not left open**: the spec recommends and the human disposes.

**Flagged, not built (v1 restraint).** No GC rule consumes `REFLECTION_STAGING.md` (auto-clear /
staleness), and no HARNESS constraint governs it. A staleness/auto-clear GC rule is **premature
before usage data** (cf. AGENTS.md "no threshold before data"); if mining proves valuable and
the staging file is found to linger, a future slice can add one. Flagged here so the absence is
deliberate, not an oversight.
