# Spec: Reflection log fragment migration

**Date**: 2026-06-15
**Author**: Russ Miles + assistant
**Driving signal**: Issue #398 — `REFLECTION_LOG.md` is a single,
shared, append-only file. Every reflection PR appends at the same EOF
location, so any two PRs cut from the same base conflict at the same
spot. When that conflict is resolved against an already-merged block,
a committed reflection is **silently dropped** — the PR merges green
and the entry simply isn't on `main`. Observed downstream: a merged
reflection vanished after two unrelated PRs merged minutes later. This
is an emergent property of the storage design, not a one-off mistake.
**Status**: design — awaiting user review

## Problem

`REFLECTION_LOG.md` is the append-only output of the plugin's
compound-learning loop. `/reflect` and the integration-agent append a
structured entry at the **end** of the file after each task. Because
there is one file with one append point (EOF):

1. **Maximal merge contention.** Two reflection PRs cut from the same
   base both append at EOF, so they conflict at the same spot.
2. **Silent data loss.** Default git merge raises a conflict; whoever
   resolves last can take one side and drop the other's appended block
   without noticing. Append-conflicts look like trivial "both added
   lines at the end" and are easy to mis-resolve against an
   already-merged entry.
3. **The log is the only collision path.** Any other files in the same
   PRs are distinct paths and survive the merge. The log is the single
   path every reflection PR touches, so it's the only thing that
   collides — and the lost content is *only* the reflection.

The failure is invisible: the merge is green, the deliverable files
survive, and only the reflection is gone.

## Goals

1. **Make reflection loss structurally impossible** under concurrent
   merges.
2. **Preserve the existing reader surface.** ~30 components (7 agents,
   2 Stop hooks, several commands, observability/telemetry) read
   `REFLECTION_LOG.md` today. They must keep working with no behaviour
   change.
3. **Preserve the archival mechanism** from the
   2026-04-30-reflection-log-archival-design.md spec: the `Promoted`
   line grammar, the weekly Path 1 auto-archive, the monthly Path 2
   aged-out review, and `reflections/archive/<YYYY>.md`.

## Non-goals

- Don't change the reflection entry schema (Date, Agent, Task,
  Surprise, Proposal, Improvement, Signal, Constraint, Session
  metadata, Promoted).
- Don't change the archive format or the Promoted-line grammar.
- Don't require every adopter to migrate immediately — the
  `.gitattributes` union default protects un-migrated monoliths.

## Design

Two layers, shipped together:

### Layer 1 — `.gitattributes` union-merge default (immediate safety)

Ship a `.gitattributes` default — in this repo's root and in the
`/superpowers-init` + `/harness-init` scaffolding — that applies git's
built-in **union** merge driver to the reflection surfaces:

```gitattributes
REFLECTION_LOG.md merge=union
reflections/archive/*.md merge=union
```

The union driver auto-combines *both* sides' appended lines on what
would otherwise be a conflict, so no entry can be dropped — it errs
toward keeping everything. This protects:

- the generated aggregate `REFLECTION_LOG.md` (two PRs that both
  regenerate it),
- the annual archive files (two GC runs that both append),
- **and** an un-migrated monolith in an adopter repo that hasn't
  adopted Layer 2 yet.

This layer alone removes the silent-drop failure mode without any
format change.

### Layer 2 — one-file-per-entry (structural fix)

**Source of truth becomes per-entry fragments.** Each reflection is
written as its own file:

```
reflections/active/<YYYY-MM-DD>-<slug>.md
```

containing exactly one entry body (the `- **Date**: …` block) with
**no** leading `---` separator. The `<slug>` is derived from the Task
field (lowercase, hyphen-separated, ≤6 words); same-date collisions
get a numeric suffix (`-2`, `-3`). No two reflections ever touch the
same file, so the append-contention — and the silent-drop failure
mode — disappears at the source.

**`REFLECTION_LOG.md` becomes a generated aggregate.** It is
deterministically regenerated by concatenating active fragments in
`Date` order (then filename), with the existing header comment and
`---` separators. It stays **committed** (like a lockfile) so all ~30
readers keep reading it unchanged. It carries `merge=union` (Layer 1)
so an interim messy merge is never lossy, and the next regeneration
normalises it from the fragments — the aggregate is **self-healing**
because the fragments are canonical.

```
reflections/
  active/
    2026-06-15-foo-bar.md      <- source of truth (write here)
    2026-06-16-baz.md
  archive/
    2026.md                    <- unchanged (Path 1 target)
  migration-proposals.md       <- unchanged (one-time migration record)
REFLECTION_LOG.md              <- GENERATED, committed, merge=union
.gitattributes                 <- merge=union on the surfaces above
```

`reflections/active/` is used (not flat `reflections/`) so the active
glob never picks up `archive/`, `migration-proposals.md`, or other
`reflections/` files.

## Mechanism

### Writing a reflection

`/reflect`, the integration-agent, and the ai-literacy-assessment
skill change their final step from "append to `REFLECTION_LOG.md`" to:

1. Write the entry body to `reflections/active/<YYYY-MM-DD>-<slug>.md`
   (body only, no leading `---`).
2. Regenerate the aggregate:
   `bash ai-literacy-superpowers/scripts/regenerate-reflection-log.sh`
   (fallback: regenerate by hand if the script is absent).
3. `git add reflections/active/<file> REFLECTION_LOG.md` and commit.

Validation checkpoint: read the fragment back and verify structure
against the entry template, fixing in place.

### Regeneration

New script `scripts/regenerate-reflection-log.sh` (thin wrapper over a
new `regenerate_log` helper in `reflection-log-helpers.sh`): enumerate
`reflections/active/*.md` sorted by `Date` then filename, emit the
header comment followed by each fragment body separated by `---`.
Deterministic and idempotent.

### Archival (Path 1, preserved)

`scripts/archive-promoted-reflections.sh` now iterates
`reflections/active/*.md` instead of splitting the monolith. For each
fragment with a verified `Promoted` line, it appends the entry to
`reflections/archive/<YYYY>.md` (format unchanged) and **deletes the
fragment**, then regenerates `REFLECTION_LOG.md`. The `Promoted`
grammar, RHS verification, and archive format are unchanged. The
weekly GC workflow (`/.github/workflows/gc.yml`) updates its
precondition to check `reflections/active/` and `git add`s the active
directory so fragment deletions are committed.

### One-time migration

New script `scripts/split-reflection-log.sh` (idempotent — skips if
`reflections/active/` already has fragments): split the existing
monolithic `REFLECTION_LOG.md` into one fragment per entry, preserving
any `Promoted` lines, then regenerate the aggregate. Existing
`reflections/archive/` is untouched. This repo's own
`REFLECTION_LOG.md` is migrated as part of this PR (dogfooding).

The existing `migrate-reflection-log.sh` (Promoted-tag proposals)
continues to operate against the aggregate and is unaffected.

## Acceptance scenarios

1. **No write collision.** Two reflections authored on the same day
   from the same base write distinct fragment files; merging both
   produces both fragments and a regenerated aggregate containing both
   entries — no conflict at the fragment level.
2. **Self-healing aggregate.** If two PRs both regenerate
   `REFLECTION_LOG.md` and union-merge produces out-of-order or
   duplicated header lines, the next `regenerate-reflection-log.sh`
   run reproduces the canonical aggregate from the fragments.
3. **Readers unchanged.** Every existing reader of
   `REFLECTION_LOG.md` continues to read the same aggregate with the
   same content; no reader is modified.
4. **Archival preserved.** A fragment tagged with a verified
   `Promoted` line is moved to `reflections/archive/<YYYY>.md` by the
   weekly Path 1 run, the fragment is deleted, and the aggregate is
   regenerated without it.
5. **Migration is lossless and idempotent.** Splitting the existing
   monolith yields one fragment per entry; re-running the split is a
   no-op; the regenerated aggregate contains every original entry.
6. **Union default ships.** A fresh `/superpowers-init` scaffolds
   `reflections/active/`, the `.gitattributes` union rules, and a
   generated `REFLECTION_LOG.md`.

## Functional requirements

- **FR1** Reflections are written as `reflections/active/<date>-<slug>.md`
  fragments, one entry per file, no leading separator.
- **FR2** `REFLECTION_LOG.md` is a deterministically regenerated,
  committed aggregate of active fragments in Date order.
- **FR3** A `.gitattributes` default applies `merge=union` to
  `REFLECTION_LOG.md` and `reflections/archive/*.md`, shipped in this
  repo and in init scaffolding.
- **FR4** Path 1 archival operates on fragments: move promoted
  fragments to the annual archive, delete them, regenerate the
  aggregate. Promoted grammar and archive format unchanged.
- **FR5** A one-time, idempotent migration splits an existing monolith
  into fragments and regenerates the aggregate, preserving Promoted
  lines and the existing archive.
- **FR6** No existing reader of `REFLECTION_LOG.md` changes behaviour.

## Risks and mitigations

- **Interim union-merge produces a cosmetically messy aggregate.**
  Mitigated by self-healing regeneration (scenario 2) — fragments are
  canonical.
- **Large file count after migration.** One fragment per historical
  entry. Acceptable: fragments are small, and Path 1 archives promoted
  entries out of `active/` over time.
- **Adopters who don't migrate.** Layer 1's union default protects the
  monolith immediately, independent of Layer 2.
