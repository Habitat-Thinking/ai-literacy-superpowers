- **Surprise**: One pattern bit *twice* in the same session and is worth recording. Doing `git mv` to relocate files, then editing the *content* of the moved files, then `git commit`-ing with a *narrowly-staged* set (e.g. `git add specific/files`) drops the in-file edits — they sit unstaged in the working tree while the rename-only commit lands on main. It happened once at PR #286 (rename to `tdad_tests/`) → #287 (cleanup of the in-file references), and again at PR #289 (move to Layer 0) → #290 (path-fix recovery). Both times the merged main was *broken* until the recovery PR landed: in #286's case stale comments and a stale README header referenced "tests-external"; in #289's case the bash scripts looked for their script targets at a path that did not exist, so Layer 0 ran exit-1 on every script. The bug is invisible to `git status` once you've staged the rename — the rename shows as a renamed-staged short-format entry, the in-file edit shows as an unstaged-modified entry next to it, and a subsequent narrow `git add` followed by `git commit` only captures what was named explicitly. The lesson is mechanical: **for any change that combines a `git mv` with content edits, either `git add -A` before `git commit`, or run `git diff --cached --stat` before committing and verify every file you edited is in the cached set.** `git commit -a` is a partial recipe but only stages tracked-file modifications, not new files, so it is not universal — the staged-stat verification is.
- **Proposal**: Add to AGENTS.md GOTCHAS — "Rename + edit gotcha: when a single logical change combines a `git mv` with content edits to the moved files, narrowly-staged `git commit` (`git add path1 path2 && git commit`) silently drops the in-file edits and ships only the rename. Verify before committing: `git diff --cached --stat` should list every file you intended to modify. Caught twice in one session (PR #286 → #287, PR #289 → #290); both times required a second PR to recover what should have been one PR." Tier: GOTCHA (a recurring footgun, not an architectural decision). Authoring deferred to a future AGENTS.md curation pass; this entry records the pattern.
- **Improvement**: One workflow rule that would have caught both occurrences: after staging a rename-and-edit change, default to `git diff --cached --stat | head` as the pre-commit check. If the cached stat shows only renames and no modifications for files I know I edited, the in-file edits aren't staged. Faster than re-reading the commit message, quicker than running tests. Worth folding into commit-related workflow notes if/when this recurs once more — second instance promoted; third instance constrains.
