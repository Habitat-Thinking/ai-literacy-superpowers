- **Proposal**: WORKFLOW (one for AGENTS.md ARCH_DECISIONS) — **Audit-after-ship for new commands.** When shipping a new command, skill, or agent that consolidates or replaces existing functionality, do not stop at editing the most-obvious docs (the new how-to page, the immediate cross-references). Grep the full `docs/plugins/<plugin>/` tree for every reference to the OLD commands or skills the new one composes/replaces, and update those references in the SAME PR. The "See also" callout pattern is too weak when the new thing is meant to be the canonical entry point — older pages must explicitly frame the older commands as primitives or alternatives, not as first-class equivalents. The PR #257 → #258 cycle was the surfacing instance; the new HARNESS.md constraint *Docs propagation when shipping new commands* (effective 2026-05-07) encodes the rule for future feature PRs. Worth promoting to AGENTS.md ARCH_DECISIONS once the constraint has caught its first real instance.
