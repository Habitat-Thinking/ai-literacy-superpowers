- **Surprise**: Two. **(a) The "Verify new components have a TDAD scenario" gate requires a scenario for EVERY new component type, not just the ones the spec's artefact list named.** The spec's artefact 8 enumerated only agent + hook scenarios, so the first implementation shipped scenarios for `agents/reservoir-warden` and `hooks/reservoir-check` but not for `skills/cognitive-reservoir` or `commands/reservoir`. CI failed with "No scenario directory at tdad_tests/scenarios/{commands/reservoir,skills/cognitive-reservoir}/" — the gate derives the required set from the *added component files in the diff*, independently of the spec's artefact list. Fixed by adding a structural scenario for each. The spec's artefact list is a guide, not the gate's source of truth. **(b) The "Shell scripts use strict mode" PR constraint checks the FIRST 15 LINES (`head -15 | grep "set -euo pipefail"`), not mere presence anywhere in the file.** The hook *had* `set -euo pipefail`, but a ~20-line explanatory header comment pushed it to line 22, so the constraint failed even though the script was correct. Fixed by moving `set -euo pipefail` to line 3 (right after the shebang and a one-line comment) with the longer header below it. The check is positional, not semantic.
- **Proposal**: Two, both for humans to decide. **(1)** A line in the `component-design-with-tdad` skill (or AGENTS.md): when a feature adds multiple component *types*, scope one TDAD scenario per new component — skill, agent, command, AND hook — up front; the "new components ship with a scenario" gate enumerates from the diff, so trusting a spec's artefact list to be exhaustive risks red CI. (Adjacent enabling fix already landed this session: the TDAD runner gained `list_hooks` so `component_type: hook` scenarios resolve a real component.) **(2)** An AGENTS.md GOTCHA: `set -euo pipefail` must sit within the first 15 lines of any `.sh` file because the "Shell scripts use strict mode" constraint greps `head -15`; put it right after the shebang with any long header comment below it.
- **Improvement**: Two. **(1)** When a spec adds components of more than one type, generate the per-component scenario stubs as part of authoring the components, not as an afterthought keyed off the spec's artefact list — the gate's required set = the added-component files in the diff. **(2)** For new shell scripts in this repo, write `set -euo pipefail` as the third line by default; treat the strict-mode constraint as positional. Both surprises were caught by *existing* gates on the first CI run and fixed in one follow-up commit — the gates worked; the lesson is to know their exact mechanics before the first push.
