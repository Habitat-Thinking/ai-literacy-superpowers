- **Task**: Amended the command-TDAD-testing design spec (PR #293, #298 amendment) after the user surfaced a buried conflation in the original Option C-direct framing. The original spec said "the markdown command becomes glue between the user and the helper" — implying helpers ship in `ai-literacy-superpowers/scripts/` and commands invoke them at runtime. The amendment supersedes that with Option I: helpers stay in `tdad_tests/spike_helpers/` as test-stage code; command markdowns continue as prose-driven instructions to the model.
- **Surprise**: The user's question was four words long: *"Why are the helpers packaged into the plugin itself?"* And it pointed at a cost the spec had completely failed to engage with. The plugin today ships shell scripts only — zero language-runtime requirements beyond bash. Promoting Python helpers into the plugin's `scripts/` would add a hard Python 3.11+ dependency on every consumer machine. The original spec's framing slid past that column of the trade-off table entirely. What's actually worth recording is *how* the omission happened. Two distinct decisions had been collapsed into one: **(a)** can a P-command's deterministic core be extracted into Python and tested with fixtures? — yes, and Phase 2 (PR #295) demonstrated this. **(b)** Should that extracted logic ship inside the plugin and be invoked at runtime? — a different question entirely, with different stakes (test coverage vs plugin distribution). The original spec answered (a) explicitly and (b) implicitly. Implicit answers to design questions are the dangerous kind: the reasoning never gets written down because the question never gets asked, so the answer isn't open to challenge. The user's question opened it; once opened, the right answer was obvious within a few minutes. The lesson: when a design recommendation has a "therefore X" that crosses a major architectural boundary — package contents, language runtimes, distribution shape — the "therefore" is not implicit, it is its own decision, and it should get its own row in the trade-off table.
- **Proposal**: Add to AGENTS.md GOTCHAS — "Conflated-decision gotcha: when a design spec proposes extracting logic for testing, it almost always carries a follow-on question — *should the extracted logic also ship as production code?* These are different decisions with different stakes. The first is about test coverage; the second is about plugin distribution / language runtimes / consumer friction. The spec must answer both explicitly. If the second answer is implicit, the spec has a hidden recommendation and should be amended before merge. Spec authors should add a 'Distribution-cost considered?' check to their pre-submit pass: for every architectural recommendation that affects what ships, did the spec name the cost?" Tier: GOTCHA (a recurring spec-authoring footgun that can hide for the lifetime of a spec until someone asks the simple question).
- **Improvement**: Two improvements, each addressing a different point in the lifecycle. **(1) Spec-author pre-submit check** — before opening a design-spec PR, run a self-review pass that asks: *for each architectural recommendation, what is the cost the spec has not weighed?* If "this would change what ships in the plugin" is one of the costs, it must appear explicitly in the trade-off discussion. The diaboli skill is the natural home for this check; it currently focuses on adversarial review of the proposed solution, not on review of what trade-offs have been omitted. **(2) Reviewer pre-merge check** — when reviewing a design spec, ask the same question on the way in. The user's *"why are the helpers packaged into the plugin itself?"* was that question, asked late. If a reviewer had asked it during PR #293's review window, the original spec would have shipped with Option I rather than amended afterwards. Worth folding into the review checklist for design-spec PRs in the same shape: *what trade-offs has the spec not weighed?*
