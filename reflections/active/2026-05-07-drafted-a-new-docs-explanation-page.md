- **Surprise**: Two related friction points around the spec-first CI check on a docs-only PR. **(a) CLAUDE.md's "Cross-Repo Spec-First Discipline" section only documents the `cross-repo` exemption explicitly**, but the workflow at `.github/workflows/spec-first-check.yml` accepts five labels (`bug`, `fix`, `chore`, `maintenance`, `cross-repo`) and three branch prefixes (`fix/`, `chore/`, `cross-repo/`). For a pure docs addition with no upstream spec — neither a feature nor a cross-repo sync — the right exemption is `chore`, but you have to read the workflow to know that. The discovery cost was a failed CI run + a label edit + a re-trigger commit. **(b) After adding the `chore` label to the open PR, the failed spec-first check did not re-run.** GitHub Actions' default `pull_request` event fires on `opened`, `synchronize`, and `reopened` only — not on `labeled`. `gh workflow run` triggered the workflow against `workflow_dispatch` but produced a separate run record that did not update the PR's check status. Pushing an empty commit fired `synchronize`, which re-ran all PR checks against the now-labelled PR and picked up the exemption. The fix worked but cost an extra commit on the branch (cleaned up by squash-merge, but the operational friction was real).
- **Proposal**: WORKFLOW (one for AGENTS.md GOTCHAS / STYLE) — **two reusable lessons.** (1) For docs-only additions outside the plugin directory with no upstream spec, the `chore` label is the correct spec-first exemption — CLAUDE.md only documents `cross-repo` explicitly, but the workflow honours `bug | fix | chore | maintenance | cross-repo`. Apply at PR creation time per the existing "Label PRs at creation time" constraint, not after the check has failed. (2) When an exemption label is added to a PR after a check has failed, the PR's check status will not refresh on its own — push an empty commit (`git commit --allow-empty -m "..."`) to fire `synchronize` and re-evaluate. `gh workflow run` against a `workflow_dispatch` trigger creates a separate run record that does not update the PR's checks UI. Worth promoting to AGENTS.md GOTCHAS only if (b) recurs — the `--label chore` pattern at PR creation should prevent (b) from arising in the first place. First instance; not yet at the recurring threshold.
- **Improvement**: Two one-off tasks would prevent this friction next time. (1) **Expand CLAUDE.md's "Cross-Repo Spec-First Discipline" into a more general "Spec-First Exemptions" section** listing all five exemption labels with one-sentence guidance on which to use for which kind of change (docs → `chore`, bug fix → `fix`, maintenance/refactor → `maintenance`, cross-repo sync → `cross-repo`). The current section's narrow framing made the answer harder to find than it needed to be. (2) **Add `labeled` and `unlabeled` to the `pull_request` trigger types in `.github/workflows/spec-first-check.yml`** so adding an exemption label after a failing run automatically re-evaluates the check. The same change would benefit any other workflow that branches on PR labels (the `gc.yml` and `enforce-pr-constraints.yml` workflows are candidates worth inspecting). Both fixes are out of scope for this reflection's PR; they are issues / follow-up work.
