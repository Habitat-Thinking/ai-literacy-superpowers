PR #306 merged green on PR-time CI but the post-merge Pages deploy aborted in `mkdocs build --strict` mode because eight docs-internal links were written relative to `docs/` instead of relative to the source file's directory (`docs/contributing/`). - **Surprise**: The PR-time CI suite has no `mkdocs build --strict` step. The strict build is exactly the verification that catches broken-link errors, but it only runs in `pages.yml` on push-to-main as part of the deploy. So the entire shape of "main is broken until we ship a fix" was structurally available: any PR that changes `docs/**` and breaks the build will land on main before anyone notices. Three things are worth recording. **(a) markdownlint is a style linter, not a link validator** — running `npx markdownlint-cli2` locally before pushing felt like docs validation but did nothing for relative-path correctness. The mental model "I lint, therefore I'm covered" was wrong. **(b) MkDocs resolves links from the source file's directory, not from `docs/` root** — obvious in hindsight but invisible without the strict build telling you. The relative-link mental model has to match the build's mental model, and the only way to verify the match is to run the build. **(c) This is the *second* "broken-main + recovery PR" pattern in the same week** — PR #286 → #287 and PR #289 → #290 were the rename+edit gotcha (different mechanism, same shape). Two structurally identical bugs in one week is a signal that the PR-time verification suite has gaps the post-merge environment fills, and the gaps are observable: any time the post-merge environment runs a check the PR doesn't, that gap is at risk of breaking main.
- **Proposal**: Add to AGENTS.md GOTCHAS — "Docs-link relative-path gotcha: when adding a docs page that links to other docs pages, MkDocs resolves links from the source file's directory, not from the docs root. A link like `plugins/foo.md` from `docs/contributing/index.md` resolves to `docs/contributing/plugins/foo.md`, not `docs/plugins/foo.md`. The strict build catches this; markdownlint does not. Pre-push: run `python3 -m mkdocs build --strict` before opening a docs PR." Tier: GOTCHA. Also worth noting the broader pattern: the PR-time CI suite should mirror the verification the post-merge environment runs, so that any failure shape that can break main is observable on the PR.
- **Improvement**: Add `mkdocs build --strict` as a PR-time CI gate, mirroring what the Pages deploy already runs post-merge. The same workflow file shape as `pages.yml`'s build job, minus the deploy step. Done in this PR via a new `.github/workflows/docs-build-check.yml` and a corresponding HARNESS.md constraint. Run cost: ~30s per PR that touches `docs/**` or `mkdocs.yml`. Run benefit: catches every broken-link, missing-page, and bad-frontmatter error before merge instead of after.
